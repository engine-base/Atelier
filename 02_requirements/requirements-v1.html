プロジェクト概要
1-1. 目的・背景

AI社員（COO／部署リーダー／専門メンバーの三階層）が常駐し、ヒアリングから納品まで開発プロジェクトの全工程を一元管理する SaaS を構築する。クライアントはAIと対話するだけでプロジェクトを完走でき、運営者（個人／受託会社）は 1人で同時10案件以上を回せる属人性の低い体制 を実現する。最終的に SaaS として広く提供し、2-3年後の事業売却（〜10億円規模）を視野に入れる。

1-2. 現状の課題
現状の課題影響
Claude Code等のAIは「実行」は強いが「管理・全体統括・品質担保」が弱いサービス品質に達するプロダクトを生むには結局人手が必要
タスク分解の粒度最適化が難しい実装漏れ・手戻り発生、タスクが無駄になる
AI実装が「楽な実装に逃げる」「仕様漏れする」仕様徹底できず、成果物の信頼性低下
議事録・提案・見積・契約・ヒアリング・要件・実装が別ツールに分散商談→納品の速度と精度が出ない
AI成果物が Markdown 単独で構造化されていないクライアント提示時の品質に問題
既存ツール（Linear/Notion/ClickUp/Devin/Cursor/v0）は単機能特化「AI組織×全工程×Claude Code×仕様徹底」を満たすものがない

機能要件詳細

主要機能の詳細は別添 step3_functional_spec.md および bf_feature_spec.json を参照。本書では中核機能のみ抜粋掲載する。

F-I03並列実行エンジン（最重要）Phase 3
概要
▶再生でタスク群をローカルClaude Codeに投入し並列実行（5-10並列）。バックグラウンド継続。
入力
選択タスク（依存解決後）、受入基準HTML、関連モック、仕様書
出力
実行ログ、成果物（コード/テスト/差分）、スコア、状態更新
処理
  • 依存解決→実行可能タスク抽出
  • 各タスクごとに Claude Code セッション起動
  • 受入基準/モック/spec を /goal 文として渡す
  • verification-loop / acceptance-criteria / test-verification スキルを同梱
  • Claude Code側で「実装→検証→スコア→再実装」自走
  • スコア基準達成 or 最大3回で停止、結果返却
エラー
  • セッション失敗→自動リトライ1回→失敗通知
  • 並列上限到達→キューイング
  • 認証エラー→初回CLIログイン誘導
制約
初回ターミナルCLIログイン必須／OAuth対応は Phase 5以降検証／ストリーミングはオプション（通常はBG）
F-J02仕様徹底ループ（差別化の心臓）Phase 3
概要
実装→検証→スコア評価→再実装をClaude Code側で自走。無限ループ防止のスコア閾値・最大再試行制御を内蔵。
スコア計算
total_score = AC達成率 × 0.6 + テスト成功率 × 0.3 + 検証AI評価 × 0.1
段階制御
  • ≥ 95% → 自動承認、次タスクへ
  • 80-94% → 人間承認ゲート（Inbox）
  • < 80% → 強制人間介入
再試行制限
最大3回。以上は人間介入
スキル統合
verification-loop / acceptance-criteria / test-verification / quality-gate
F-K01ナレッジ管理基盤（2層構造）Phase 4
概要
アカウント単位の共通＋AI社員別の2層ナレッジ。プロジェクト跨ぎで再活用。
共通
ユーザー方向性 / 業界傾向（3案件以上で自動抽出）/ プロジェクト横断テンプレ
社員別
  • ワンダ → 参考デザイン / カラー傾向 / フォント選好
  • トニー → 過去提案テンプレ / 成約パターン
  • ストレンジ → 採用アーキ / 失敗パターン
  • ヴィジョン → 検証チェックリスト
  • ティチャラ → 全社員のナレッジ統括
参照
各社員は自分のナレッジ優先＋共通も参照可
書込
AI候補生成→ユーザー承認で書込み
整理
AI自動でカテゴリ・タグ・重複統合（夜間cron）
検索
ベクトル＋キーワードハイブリッド、Voyage Reranker
F-I05/goal駆動実行モードPhase 3
概要
タスクカード→ゴール文自動生成→Claude Code /goal 投入→自走実行→結果収集
ゴール生成
タイトル＋受入基準＋依存タスク＋関連モックパスから自動生成
利点
Claude Codeの自走能力を最大活用、ツール側はゴール定義と結果収集に専念

6-4. 主要ユーザーフロー
  1. UF-1：サインアップ → WS作成 → デフォルト10名AI社員配属 → プロジェクト作成 → ジャービスが「どこから始める？」提案 → ヒアリングor議事録アップロード
  2. UF-2：ピーターがタスク分解 → かんばん配置 → ▶再生 → ローカルClaude Code起動 → /goal自走 → スコア評価で承認/再実装/介入 → バックグラウンド継続
  3. UF-3：オーナーがクライアント招待発行 → メール送信 → クライアントが招待リンクからログイン → 全フェーズ閲覧 → コメント投稿 → AI修正提案 → オーナー承認 → 反映
  4. UF-4：プロジェクト進行中ティチャラがナレッジ候補抽出 → ユーザー承認 → 書込＋埋め込み生成 → 別プロジェクトで自動参照 → 3件以上で業界ナレッジ昇格
  5. UF-5：MCPトークン発行 → Claudeデスクトップに設定 → AI社員呼出 → ナレッジ参照→生成→返却
