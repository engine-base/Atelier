# T-F-40: knowledge_nodes scope=employee_specific 細粒度 RLS (将来用)

## Purpose
T-D-18 で配置した `knowledge_nodes` RLS は **workspace member 全員可視** を採用
した (entities.json E-018 仕様に scope=employee_specific の閲覧者制限が
明示されていないため)。

将来「owner_employee に紐づかないユーザーは employee_specific knowledge を
見られない」という細粒度制御が必要になった場合、本タスクで以下を実装する:

## 設計
- `public.user_can_see_knowledge_node()` helper を再定義:
  - scope=employee_specific の場合、owner_employee_id が指す ai_employee の
    workspace の owner role である、または owner_employee_id 自身に紐づく
    user_id である場合のみ true を返す
  - scope=common の場合は従来通り workspace member 全員可視

## Tier 1 — Structural
- helper function `user_can_see_knowledge_node` が新しいロジックに更新される

## Tier 2 — Functional (EARS)
- UBIQUITOUS: idempotent
- UNWANTED: workspace member だが owner_employee に紐づかない user が
  employee_specific knowledge を query → 0 rows

## Tier 3 — Regression
v3-gate.yml の 13 gate 全 PASS。
scripts/verify_rls_isolation.py に employee_specific 越境試験を追加。

## 依存
- T-D-18 (基本の RLS が配置済)
- T-D-22 (致命級 R-T08 が解決済で JWT 経路完全分離が機能している)

## 起票時の判断
本タスクは T-D-18 の design_decisions_inline で「将来必要なら起票」と明示
された follow-up。現状の Atelier ユースケースでは workspace member 全員可視で
要件を満たすため、Wave 2 に配置 (急がない)。
