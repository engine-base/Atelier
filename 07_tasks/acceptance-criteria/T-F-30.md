# T-F-30 CI Gate #11 — PR scope guard

## Purpose

PR の touched files が `tickets.json#T-X-Y.files_changed_predicted (new ∪ modify)`
の部分集合であることを CI で機械的に検証する。これにより「ついでに修正」「placeholder
で別ファイルを直す」等の境界違反を構造的に防ぐ。

## Trigger

ブランチ名 `feat/t-x-y-...` から task ID を抽出して該当 task の boundary をチェックする。

## Behavior

1. `git diff --name-only origin/main..HEAD` で touched files を列挙
2. ブランチ名から task ID を抽出
   - 正規表現: `^feat/(t-[a-z]-[0-9]+)-.*$` (case insensitive)
   - 例: `feat/t-f-30-...` → `T-F-30`
3. tickets.json から該当 task の `files_changed_predicted.new ∪ modify` を取得
4. touched files の中で `new ∪ modify` に含まれないものがあれば **FAIL**
5. ブランチ名から task ID が抽出できない場合は **warning のみで pass** (fix/* / chore/* PR 等)

## Exemption

以下のファイルは scope guard の対象外:
- `07_tasks/acceptance-criteria/T-X-Y.md` (自タスクの AC document)
- `pnpm-lock.yaml` / `uv.lock` / `package-lock.json` (lockfile は dep 変更で自動更新)

## Failure Modes

| 状態 | 挙動 |
|---|---|
| 境界違反検出 | FAIL + 違反 file 一覧を `::error::` で出力 |
| task ID 抽出不可 | warning + pass (fix/chore PR 等) |
| task が tickets.json に無い | FAIL ("unknown task ID") |

## Implementation Note

`v3-gate.yml` に `gate-11-scope-guard` job を追加。inline python heredoc で実装し、
他ファイルへの分割はしない (T-F-30 の files_changed_predicted は v3-gate.yml のみ)。

## Acceptance

3-tier AC の全項目が PASS することで T-F-30 完了。本タスク自身も Gate #11 の対象であり、
本 PR の touched files (`07_tasks/acceptance-criteria/T-F-30.md` + `.github/workflows/v3-gate.yml`)
が `T-F-30.files_changed_predicted` の new ∪ modify に含まれることをセルフテストする。
