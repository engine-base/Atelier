Supabase RLS パターン

2026-05-12 作成 · 2026-05-18 更新
T
ティチャラが整理 参照 12 回
supabase rls multi-tenant security

Atelier では、workspace_id + RLS でマルチテナント分離を実現する。クライアント招待 は別 JWT 経路（role=client_portal）で完全分離する。

基本構造

  • 通常ユーザーは auth.uid() + workspace_memberships で判定
  • クライアントは JWT claim の invitation_token_hash + project_id で判定
  • service_role は FastAPI 内のシステム書込専用
重要：クライアント JWT は workspace_memberships テーブルを一切参照しない。これで R-T08 致命級リスク（クライアント漏洩）に対策できる。

predicate ライブラリ

  • fn_is_workspace_member(ws_id)
  • fn_is_workspace_owner(ws_id)
  • fn_is_client_invitation_valid(invitation_id)
  • fn_is_admin()

過去案件での適用

小松様 EC モール統合プロジェクトで同じパターンを採用済み。クライアント招待での閲覧専用 UI と組み合わせて、漏洩ゼロで運用中。

関連：LangGraph 人間承認ループ · 監査ログ設計
