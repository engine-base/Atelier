01
全体方針

モジュラーモノリス × 2-tier：Next.js（フロント+BFF）と Python FastAPI（AI処理層）の 2 サービス構成。共通DBは Supabase Postgres（Tokyo region）。1人運用＋3ヶ月期限＋並列5-10AIタスクという制約に対する現実解。

マイクロサービス分割は1人運用で不可能、完全モノリスは AI 機能を犠牲にする。AI 処理層を独立させることで、Pythonエコシステム（Anthropic SDK / Agent SDK / LangGraph / LlamaIndex / Cognee）を100%活用しつつ、Next.js でエディトリアル品質の UI を実現する。

4-2. Access Control (RLS)

致命級リスク R-T08（クライアント漏洩）対策として、通常ユーザー JWT とクライアント招待 JWT を 完全分離。全27テーブルに per-entity policy を定義し、Foundation Gate #4 で網羅性を CI 検証。

predicates_lib:
  self_only         : user_id = auth.uid()
  ws_member         : workspace_id IN (SELECT ... FROM workspace_memberships WHERE user_id = auth.uid())
  ws_owner          : ws_member AND role = 'owner'
  project_member    : project_id IN (SELECT id FROM projects WHERE ...)
  client_via_invite : invitation_token_hash 一致 AND expires_at > now()
  admin_only        : raw_app_meta_data->>'role' = 'admin'
  service_role      : auth.role() = 'service_role'

インフラ・デプロイ構成
環境用途ホスティング
localローカル開発Supabase CLI Docker
previewPR ごと自動生成Vercel Preview + Supabase preview branch + Fly.io review
stagingリリース前検証Vercel staging + Supabase staging + Fly.io staging Tokyo
production本番Vercel + Supabase production + Fly.io Tokyo

環境間データ分離：staging と production は別 Supabase project（DB/Auth/Storage すべて別）。preview は staging 読取専用クローン or 空 DB。本番データは preview / staging に流入しない（個情法対応）。

Foundation Phase Gate（8 merge gate）

並列5-10 開発で main を守るための CI Gate。.github/workflows/v3-gate.yml に実装。

Retry プロトコル：最大3回連続失敗で human エスカレーション。retry 時に前回ログを Claude Code に渡してコンテキスト引き継ぎ、別 worktree で起動。

並列開発戦略
workflow         : trunk-based + worktree
branch_naming    : <agent>/<task_id>   例: claude/T-042
worktree_pattern : $REPO_ROOT/../worktrees/<task_id>
merge_method     : squash + auto-merge (CI 全 gate pass 時)
conflict_avoid   : task-decomposition の files_changed で事前回避
parallel_capacity: small (5-10)
