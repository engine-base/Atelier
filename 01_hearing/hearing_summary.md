# Atelierヒアリングサマリー
作成日：2026-05-18 ／ 最終更新：2026-05-18（要件定義STEP3反映）
ヒアリング対象：高本まさと（代表 / 1人開発体制 ＋ チーム）

## プロジェクト概要
- **作るもの**：AI社員が常駐し、ヒアリング〜納品まで開発プロジェクトを一元管理する統合ツール（自社サービス）
- **本質的なゴール**：1人で同時10案件以上を回せる属人性の低い開発プロジェクト統合管理SaaSを作り、2026年内に100社/100名利用、最終的に2-3年後に〜10億円規模での売却を視野
- **成功の定義（定量）**：2026年内100社/100名利用達成、SaaS課金開始、自分が同時10案件並行運用可能
- **成功の定義（定性）**：Claude以外のLLM（GPT/Gemini/ローカル）でも品質維持、属人性が低く他者でも運用・拡張・売却可能
- **リリース計画**：2026年6月ベータ開始、2026年8月本格リリース

## 優先機能（Must / 19項目）
1. ワークスペース/プロジェクトCRUD
2. 認証・課金プラン管理（Stripe、BYOK＋運営APIキーモード切替）
3. AI社員三階層（COO/リーダー/メンバー、人格カスタマイズ可・スキル中身不可）
4. スキル取り込み機構（~/.claude/skills/ からコピー、独自管理）
5. チャットUI（OSS/SDK活用、Claudeレベル品質、人間直編集即反映）
6. コンテキスト圧縮層（フェーズ間引き継ぎ用、Agent SDK標準＋LLMLingua）
7. 工程ワークフロー（ヒアリング〜納品、フェーズ切替・進捗可視化）
8. 三層出力（HTML/JSON/MD）
9. フェーズ分割管理（AI提案→ユーザー選択UI）
10. デザインモック（HTML、チャット内作成・修正）
11. タスク分解スキル統合（1-2日粒度、依存関係必須属性）
12. かんばん/リスト
13. 並列実行エンジン（5-10並列、ローカルClaude Codeブリッジ）
14. 仕様徹底ループ（受入基準HTML化＋テスト仕様＋検証AI、スコア評価で停止判定）
15. 人間承認ゲート
16. ナレッジ管理（アカウント単位、Voyage AI + pgvector + LlamaIndex + Cognee）
17. MCP連携（Claudeデスクトップ等から呼出）
18. クライアント招待モード（コメント→修正反映）
19. データ削除・退会フロー

## ナレッジ構造（追記）

アカウント単位のナレッジは **共通＋AI社員別の2層構造** で管理する。

```
[ アカウント単位ナレッジ ]
  ├─ 共通ナレッジ
  │   ├─ ユーザーの方向性・好み
  │   ├─ 業界傾向（3案件以上で自動抽出）
  │   └─ プロジェクト横断テンプレ
  │
  └─ AI社員別ナレッジ（各社員が優先参照）
      ├─ デザイン部ワンダ → 参考デザイン一覧 / カラー傾向 / フォント選好
      ├─ 営業部トニー     → 過去提案テンプレ / 成約パターン / 価格戦略
      ├─ 設計部ストレンジ → 採用したアーキテクチャ / 失敗パターン
      ├─ 検証部ヴィジョン → よくある仕様逸脱 / 検証チェックリスト
      └─ ...
  
  ※ 各社員は「自分のナレッジ」優先＋「共通ナレッジ」も参照可
  ※ プロジェクト跨ぎで自動再利用、ユーザー承認時に追加・更新
```

## リスクと対策
| リスク | 影響 | 対策 |
|---|---|---|
| 3ヶ月での全Phase完走 | High | Phase 1で本人1案件完走を最優先MVP、それ以外は柔軟圧縮 |
| 検証ループの無限化 | High | スコア閾値（受入基準95%）で停止＋人間承認ゲート |
| ローカルClaude Codeブリッジの不安定性 | High | OAuth連携可否を早期検証、ダメなら初回CLIログイン |
| コンテキスト圧縮で情報欠落 | High | HTML原本を圧縮対象外として保持・再参照可能化 |
| 依存関係漏れによるタスク失敗 | High | タスクカードにdependencies/prerequisites/blocks必須属性化 |
| プロバイダー追加時の再実装 | High | Phase 0でLLMClient抽象化レイヤー必須実装 |
| 売却時KPI軽視 | Medium | Phase 5でアナリティクス計測組込 |
